# Промпт для Codex: плагин проверки силлабуса на сабагентах

> Скопируй всё, что ниже черты, и вставь одним сообщением в Codex (CLI или app),
> запущенный в корне репозитория `sylabys`.

---

Ты строишь «плагин проверки силлабуса» на нативных сабагентах Codex. Спецификация проверки уже написана и лежит в репозитории — ты её НЕ переписываешь, а реализуешь:

- `docs/verification-algorithm.md` — тест-кейсы в 6 сьютах (STR, FMT, OP, RUP, INT, TXT), вердикты `PASS/FAIL/WARN/SKIP/NEEDS_HUMAN`, severity, JSON-формат записи отчёта (п. 8), правила арбитра;
- `docs/syllabus-standard.md` — эталонная структура силлабуса;
- `src/types/syllabus.ts` — модель данных (контракт для экстракции);
- фикстуры в корне репозитория: силлабус (`Силлабус дисциплины … .docx`), паспорт ОП (`Приложение 6 … .docx`), РУП (`08.06 РУП … .xlsx`), шаблон (`syllabus-template от ДАР.docx`).

Прочитай эти файлы ПЕРЕД началом работы. При любом конфликте между твоими идеями и спецификацией — спецификация главнее.

## Архитектура (фиксированная, не менять)

```
@sylabys-syllabus-checker Проверь силлабус <docx> через 6 субагентов
   │
   ├─ scripts/extract.py  → build/syllabus.json, build/op.json, build/rup.json
   │
   ├─ spawn_agents (ПАРАЛЛЕЛЬНО, 6 шт.):
   │     syllabus-str  → reports/str.json
   │     syllabus-fmt  → reports/fmt.json
   │     syllabus-op   → reports/op.json
   │     syllabus-rup  → reports/rup.json
   │     syllabus-int  → reports/int.json
   │     syllabus-txt  → reports/txt.json
   │
   └─ арбитр (ГЛАВНЫЙ тред, без спавна) → reports/final-report.json + .md
```

Три принципа:
1. Все DET-проверки — детерминированные Python-скрипты. LLM-агенты выполняют только LLM-тесты своего сьюта, а DET-часть получают вызовом скрипта.
2. Агенты обмениваются данными ТОЛЬКО через файлы (`build/`, `reports/`), не через память тредов.
3. Один уровень вложенности: главный тред спавнит шесть агентов, ждёт всех, затем сам играет роль арбитра. Сабагенты сабагентов не спавнят.

## Этап 0 — конфигурация сабагентов (в прошлый раз сломалось именно здесь)

Сделай и ПРОВЕРЬ, прежде чем строить остальное:

1. В `~/.codex/config.toml` добавь (не удаляя существующее):
   ```toml
   [agents]
   max_threads = 6
   max_depth = 1
   ```
2. Кастомные агенты — отдельные TOML-файлы в каталоге `.codex/agents/` В КОРНЕ ПРОЕКТА (по одному файлу на агента). В каждом файле ОБЯЗАТЕЛЬНЫ все три поля: `name`, `description`, `developer_instructions` — без любого из них агент молча не регистрируется.
3. Помни: сабагенты никогда не запускаются сами по себе. Оркестрирующий промпт обязан содержать явную команду вида «spawn agents …, wait for all». Отсутствие автозапуска — нормальное поведение, не баг.
4. Smoke-тест механизма: создай один минимальный агент, заспавни его с тривиальной задачей («прочитай README и верни одну строку») и убедись, что результат вернулся в главный тред. Если спавн не работает — ОСТАНОВИСЬ, сообщи диагностику (версия Codex, содержимое TOML, текст ошибки) и не продолжай.

## Этап 1 — детерминированное ядро (`scripts/`, Python)

1. `scripts/extract.py`:
   - силлабус `.docx` → `build/syllabus.json` строго по модели `src/types/syllabus.ts`;
   - паспорт ОП (Приложение 6) → `build/op.json`: шифр/наименование ОП, цель, программные РО1–РО10 с формулировками, по каждой дисциплине — цикл, компонент, кредиты и матрица «+» по РО;
   - РУП `.xlsx` → `build/rup.json`: часы, кредиты, периоды, кафедры, связь модуль→дисциплины.
   - Раздел, который не удалось распарсить, помечается в JSON как `"_extraction_error"` — тесты этого раздела потом получат `SKIP`, а не падение скрипта.
2. `scripts/det_checks.py --suite STR|FMT|OP|RUP|INT|TXT --build build/ --out reports/`:
   - выполняет ВСЕ тесты типа DET своего сьюта по `docs/verification-algorithm.md`;
   - печатает JSON-массив вердиктов ровно в формате п. 8 спецификации (`testId`, `verdict`, `severity`, `discipline`, `location`, `expected`, `actual`, `evidence`, `recommendation`, `confidence: 1.0`);
   - детерминирован: два прогона на одних данных дают побайтово одинаковый результат.
3. Покрой ядро pytest-тестами: синтетический мини-силлабус с заранее внесёнными ошибками (несошедшиеся часы, битый код РО, лишний «+» в матрице ОП) → ожидаемые вердикты.

## Этап 2 — шесть агентов-проверяющих (`.codex/agents/`)

Файлы: `syllabus-str.toml`, `syllabus-fmt.toml`, `syllabus-op.toml`, `syllabus-rup.toml`, `syllabus-int.toml`, `syllabus-txt.toml`. Скелет каждого:

```toml
name = "syllabus-op"
description = "Проверяет силлабус на соответствие паспорту ОП (сьют OP из docs/verification-algorithm.md). Использовать при прогоне @sylabys-syllabus-checker."
sandbox_mode = "workspace-write"
model_reasoning_effort = "high"
developer_instructions = """
Ты — агент-эксперт соответствия ОП из спецификации docs/verification-algorithm.md (сьют OP).
Алгоритм работы, строго по шагам:
1. Прочитай раздел своего сьюта в docs/verification-algorithm.md.
2. Запусти: python scripts/det_checks.py --suite OP — получишь готовые DET-вердикты.
3. Выполни сам ТОЛЬКО тесты с пометкой LLM своего сьюта, по build/*.json.
   Каждый не-PASS вердикт обязан содержать цитату-доказательство (evidence)
   и confidence; при confidence < 0.7 ставь вердикт NEEDS_HUMAN.
4. Объедини DET- и LLM-вердикты и запиши ОДИН файл reports/op.json
   (JSON-массив в формате п. 8 спецификации). Больше никаких файлов не менять.
5. Финальное сообщение — одна строка: «OP: N тестов, X FAIL, Y WARN, Z NEEDS_HUMAN».
Запрещено: редактировать силлабус и спецификацию, спавнить других агентов,
выходить за пределы своего сьюта.
"""
```

Сделай аналогично для остальных пяти, меняя сьют, имя выходного файла и роль (STR — структурный контролёр, FMT — технический контролёр, RUP — эксперт учебного плана, INT — аудитор согласованности, TXT — редактор-вычитчик). У FMT добавь в инструкции работу с docx-разметкой напрямую (шрифты, поля, track changes), у TXT — что он полностью LLM и скрипт ему не нужен.

## Этап 3 — оркестрация (`@sylabys-syllabus-checker`)

Plugin workflow `@sylabys-syllabus-checker Проверь силлабус <путь к docx> через 6 субагентов`. Текст оркестрации обязан содержать команду спавна ЯВНО, например:

```
1. Очисти build/ и reports/, запусти python scripts/extract.py <docx>.
2. Spawn six agents in parallel using spawn_agents:
   syllabus-str, syllabus-fmt, syllabus-op, syllabus-rup, syllabus-int, syllabus-txt.
   Передай каждому: пути build/ и reports/, путь к исходному docx (для FMT и TXT).
   Wait for ALL six to finish.
3. Проверь наличие всех шести reports/*.json. Отсутствующий отчёт —
   перезапусти соответствующего агента ОДИН раз; повторный сбой = инцидент в отчёте.
4. Роль арбитра выполни САМ в главном треде (НЕ спавнь седьмого агента):
   - склей каскадные замечания в одну корневую причину
     (пример: часы разошлись с РУП → закономерно упал и тест темплана INT-021);
   - отсортируй CRITICAL → MAJOR → MINOR → WARN;
   - вынеси NEEDS_HUMAN в отдельный блок «на ручную экспертизу» с цитатами;
   - вычисли итоговый статус по п. 1.3 спецификации;
   - запиши reports/final-report.json и человекочитаемый reports/final-report.md
     (структура акта — по 20-этапному алгоритму, см. п. 9 спецификации).
```

## Этап 4 — приёмочный прогон

Запусти `@sylabys-syllabus-checker` на реальном силлабусе из корня репозитория. Критерии приёмки:

- созданы все 6 `reports/*.json` и `final-report.json` + `final-report.md`;
- в каждом отчёте ≥ 1 вердикт; у каждого FAIL заполнены `expected`, `actual`, `evidence`;
- повторный запуск `det_checks.py` даёт идентичные DET-вердикты;
- арбитр действительно склеил каскад (если расхождение часов есть — оно фигурирует один раз как корневая причина);
- pytest по ядру зелёный.

Если хоть один агент не вернул отчёт — не сдавай работу: диагностируй по чек-листу (все три обязательных поля в TOML? файл лежит в `.codex/agents/`? в оркестрирующем промпте есть явное «spawn … wait for all»? не упёрся ли в max_threads?) и почини.

## Запреты

- НЕ использовать `spawn_agents_on_csv` (экспериментальный) — только обычный `spawn_agents`.
- НЕ поднимать `max_depth` выше 1 и не проектировать вложенный спавн.
- НЕ давать сабагентам действий, требующих новых approval (в неинтерактивном режиме они падают): всё необходимое должно укладываться в `sandbox_mode = "workspace-write"` без доступа в сеть.
- НЕ изменять `docs/verification-algorithm.md`, `docs/syllabus-standard.md`, `src/types/syllabus.ts` и исходные docx/xlsx.
