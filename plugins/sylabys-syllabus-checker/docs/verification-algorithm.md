# Алгоритм проверки силлабуса (спецификация тест-сьютов)

**НАО «Медицинский университет Астана»**
Логическая спецификация проверки силлабуса в стиле модульных тестов (аналог PHPUnit).
Документ — основа для будущего проверяющего плагина: несколько ИИ-агентов исполняют тест-сьюты и формируют сводный акт экспертизы.

> Этот документ описывает **что и как проверяется**. Сам плагин здесь не реализуется.

---

## 1. Концепция

Проверка устроена как прогон набора тест-сьютов над «фикстурами» (входными документами):

```
Фикстуры ──> Агент-экстрактор ──> JSON-модель силлабуса (src/types/syllabus.ts)
                                        │
        ┌───────────┬───────────┬───────┴────┬────────────┬───────────┐
        ▼           ▼           ▼            ▼            ▼           ▼
   Агент STR   Агент FMT   Агент OP     Агент RUP    Агент INT   Агент TXT
   (структура) (оформление)(сверка с ОП)(сверка с РУП)(внутренняя (качество
                                                       согласован.) текста)
        └───────────┴───────────┴────────────┴────────────┴───────────┘
                                        │
                                        ▼
                          Агент-арбитр: сводный отчёт
              (дедупликация, приоритизация, акт экспертизы)
```

### 1.1. Фикстуры (входные артефакты)

| Код | Артефакт | Роль | Если отсутствует |
|---|---|---|---|
| `F-SYL` | Силлабус (docx) — объект проверки | обязательная | прогон невозможен |
| `F-OP` | Паспорт ОП («ОПешка», Приложение 6): цель ОП, 10 программных РО, матрица «дисциплина → РО», кредиты, цикл/компонент | для сьюта OP | сьют OP → `SKIP` |
| `F-RUP` | РУП (xlsx): часы, кредиты, периоды, связь модуль→дисциплины | для сьюта RUP | сьют RUP → `SKIP` |
| `F-TPL` | Эталонный шаблон (`docs/syllabus-template.docx`) + стандарт структуры (`docs/syllabus-standard.md`) | для сьютов STR, FMT | сьюты STR/FMT → `SKIP` |

### 1.2. Типы проверок

| Тип | Кто исполняет | Пример |
|---|---|---|
| `DET` — детерминированная | код (без нейронки), агент только запускает | сумма часов, наличие раздела, сквозная нумерация |
| `LLM` — семантическая | ИИ-агент с обязательной цитатой-доказательством | «цель дисциплины не противоречит цели ОП» |

Правило: всё, что можно проверить детерминированно, **не отдаётся** нейронке. LLM-проверки обязаны возвращать `evidence` (точную цитату + локацию) и `confidence`; при `confidence < 0.7` вердикт автоматически понижается до `NEEDS_HUMAN`.

### 1.3. Вердикты и критичность

| Вердикт | Смысл |
|---|---|
| `PASS` | утверждение выполнено |
| `FAIL` | утверждение нарушено; обязательны `expected / actual / evidence` |
| `WARN` | не блокирует утверждение, но требует внимания |
| `SKIP` | проверка неприменима или нет фикстуры (с указанием причины) |
| `NEEDS_HUMAN` | LLM не уверена — на ручную экспертизу |

| Severity | Следствие |
|---|---|
| `CRITICAL` | силлабус не может быть утверждён |
| `MAJOR` | вернуть на доработку |
| `MINOR` | устранить до утверждения, не блокирует экспертизу |

Итоговый статус силлабуса: `ОТКЛОНЁН`, если есть хотя бы один `FAIL/CRITICAL`; `НА ДОРАБОТКУ`, если есть `FAIL/MAJOR`; иначе `ПРИНЯТ С ЗАМЕЧАНИЯМИ` (есть `WARN`) или `ПРИНЯТ`.

### 1.4. Формат тест-кейса

```
ТЕСТ <ID> «<название>»
Тип: DET | LLM      Severity: CRITICAL | MAJOR | MINOR
Контекст: <силлабус в целом | по каждой дисциплине | только module | только discipline>
Дано:    <какие разделы/фикстуры берём>
Когда:   <что вычисляем/сопоставляем>
Тогда:   <утверждение(я) — assert>
При FAIL: <что включить в отчёт>
```

Контекст «по каждой дисциплине» — аналог `@dataProvider disciplines` в PHPUnit: для дисциплинарного силлабуса тест выполняется 1 раз, для модульного — N раз (по каждой дисциплине из `disciplines[]`), и каждый прогон получает свой вердикт.

## 2. Сьют STR — Структура документа

Агент: **Структурный контролёр**. Источник истины: `docs/syllabus-standard.md`, разделы 0–15 + Приложения 1–2.

```
ТЕСТ STR-001 «Тип силлабуса определён»
Тип: DET      Severity: CRITICAL
Контекст: силлабус в целом
Дано:    титул и раздел 0
Когда:   определяем тип (module | discipline) и считаем |disciplines[]|
Тогда:   assertNotNull(type)
         И (type == 'discipline' → count(disciplines) == 1)
         И (type == 'module'     → count(disciplines) >= 2 И заполнен ModuleInfo)
При FAIL: указать противоречие (например: заявлен модуль, но дисциплина одна)
```

```
ТЕСТ STR-002 «Все обязательные разделы присутствуют»
Тип: DET      Severity: CRITICAL
Контекст: силлабус в целом
Дано:    эталонный перечень разделов: титул; описание; преподаватели; цель и краткое
         содержание; РО; ПН; тематический план; график СРОП; политика; оценивание;
         литература; глоссарий; Приложение 1 (КИС); Приложение 2 (КО); согласование
Когда:   сопоставляем заголовки/разделы силлабуса с перечнем
Тогда:   assertEmpty(списокОтсутствующихРазделов)
При FAIL: перечислить отсутствующие разделы
```

```
ТЕСТ STR-003 «Порядок разделов соответствует шаблону»
Тип: DET      Severity: MINOR
Контекст: силлабус в целом
Дано:    фактическая последовательность разделов и эталонная из шаблона
Когда:   сравниваем последовательности
Тогда:   assertSame(эталонныйПорядок, фактическийПорядок)
```

```
ТЕСТ STR-004 «Повторение разделов в модульном силлабусе»
Тип: DET      Severity: MAJOR
Контекст: только module
Дано:    type == 'module', disciplines[] = N
Когда:   проверяем кратность разделов
Тогда:   разделы «Описание», «РО», «ПН», «Тематический план», «Оценивание»,
         «Литература», Приложения 1–2 присутствуют по каждой из N дисциплин;
         разделы «Титул», «Преподаватели», «Цель и содержание», «График СРОП»,
         «Политика», «Глоссарий», «Согласование» — ровно один раз
При FAIL: указать, для какой дисциплины раздел отсутствует / какой раздел продублирован
```

```
ТЕСТ STR-005 «Нет посторонних разделов»
Тип: DET      Severity: MINOR
Контекст: силлабус в целом
Дано:    перечень разделов силлабуса
Когда:   ищем разделы вне эталонной структуры
Тогда:   assertEmpty(посторонниеРазделы) — иначе WARN с перечнем
```

## 3. Сьют FMT — Оформление

Агент: **Технический контролёр** (работает с docx напрямую, до экстракции).

```
ТЕСТ FMT-001 «Шрифт и кегль»
Тип: DET      Severity: MINOR
Дано:    стили документа (word/styles.xml и прямое форматирование)
Тогда:   assertEquals('Times New Roman', шрифт) И assertEquals(12, кегль)
         для основного текста (заголовки/таблицы — по шаблону)
При FAIL: перечислить фрагменты с другим шрифтом/кеглем
```

```
ТЕСТ FMT-002 «Межстрочный интервал»
Тип: DET      Severity: MINOR
Тогда:   assertEquals('одинарный', интервал основного текста)
```

```
ТЕСТ FMT-003 «Поля страницы»
Тип: DET      Severity: MINOR
Тогда:   верхнее = 1,5 см; нижнее = 1,5 см; левое = 2,5 см; правое = 1 см
```

```
ТЕСТ FMT-004 «Нет следов редактирования»
Тип: DET      Severity: CRITICAL
Дано:    docx-разметка
Тогда:   assertFalse(естьTrackChanges) И assertFalse(естьКомментарии)
         И assertFalse(естьЦветныеВыделения-маркеры)
При FAIL: перечислить места с правками/комментариями/выделениями
```

## 4. Сьют OP — Сверка с ОП («ОПешкой»)

Агент: **Эксперт соответствия ОП**. Фикстура `F-OP` — паспорт ОП (Приложение 6): шифр и наименование ОП, цель ОП, программные РО1–РО10 (с формулировками), «Сведения о дисциплинах» — по каждой дисциплине: наименование, краткое описание, цикл, компонент, кредиты и матрица «+» по РО1–РО10.

```
ТЕСТ OP-001 «Шифр и наименование ОП»
Тип: DET      Severity: CRITICAL
Контекст: силлабус в целом
Дано:    титул и раздел «Описание» силлабуса; паспорт ОП
Тогда:   assertEquals(OP.шифр, силлабус.шифрОП)          // напр. 6В10123
         И assertEquals(OP.наименование, силлабус.наименованиеОП)  // «Медицина»
```

```
ТЕСТ OP-002 «Дисциплина существует в ОП»
Тип: DET+LLM      Severity: CRITICAL
Контекст: по каждой дисциплине
Дано:    наименование дисциплины в силлабусе; перечень «Сведения о дисциплинах» ОП
Когда:   ищем дисциплину в перечне (точное совпадение, при расхождении —
         LLM-сопоставление с порогом уверенности)
Тогда:   assertNotNull(соответствующаяСтрокаОП)
При FAIL: дисциплины нет в ОП — блокирующее замечание
```

```
ТЕСТ OP-003 «Цикл и компонент»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Тогда:   assertEquals(OP.строкаДисциплины.цикл, силлабус.цикл)        // ООД/БД/ПД
         И assertEquals(OP.строкаДисциплины.компонент, силлабус.компонент) // ОК/ВК/КВ
```

```
ТЕСТ OP-004 «Кредиты дисциплины»
Тип: DET      Severity: CRITICAL
Контекст: по каждой дисциплине
Тогда:   assertEquals(OP.строкаДисциплины.кредиты, силлабус.кредиты)
```

```
ТЕСТ OP-005 «Маппинг РО: нет лишних связей»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Дано:    маппинг РО дисциплины → программные РО (раздел 5 силлабуса);
         строка матрицы «+» этой дисциплины в ОП
Когда:   собираем множество A = {программные РО, заявленные в силлабусе},
         множество B = {РО с «+» в матрице ОП}
Тогда:   assertTrue(A ⊆ B)
При FAIL: перечислить РО из A \ B («силлабус заявляет связь, которой нет в ОП»)
```

```
ТЕСТ OP-006 «Маппинг РО: полное покрытие матрицы»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Когда:   те же множества A и B
Тогда:   assertTrue(B ⊆ A)
При FAIL: перечислить РО из B \ A («ОП требует формировать РО, не покрытый силлабусом»)
```

```
ТЕСТ OP-007 «Формулировки программных РО не искажены»
Тип: DET      Severity: MAJOR
Контекст: силлабус в целом
Дано:    цитаты программных РО в силлабусе (если приводятся); эталон из ОП
Тогда:   assertEquals(эталоннаяФормулировка, цитата) — посимвольно после нормализации
         пробелов; расхождение = FAIL с diff
```

```
ТЕСТ OP-008 «Краткое содержание соответствует описанию из ОП»
Тип: LLM      Severity: MAJOR
Контекст: по каждой дисциплине
Дано:    краткое содержание (раздел 4/2) силлабуса; краткое описание дисциплины из ОП
Когда:   семантическое сопоставление: тот же предмет, та же область, нет подмены тематики
Тогда:   assertSemanticallyConsistent(содержаниеСиллабуса, описаниеОП)
При FAIL: процитировать оба фрагмента и назвать расхождение
```

```
ТЕСТ OP-009 «Цель не противоречит цели ОП»
Тип: LLM      Severity: MAJOR
Контекст: силлабус в целом (цель модуля или дисциплины)
Тогда:   цель силлабуса конкретизирует (не противоречит и не выходит за рамки) цель ОП
```

```
ТЕСТ OP-010 «Язык преподавания допустим в ОП»
Тип: DET      Severity: MINOR
Контекст: по каждой дисциплине
Тогда:   assertContains(OP.языкиОбучения, силлабус.языкПреподавания)
```

## 5. Сьют RUP — Сверка с РУП

Агент: **Эксперт учебного плана**. Фикстура `F-RUP`.

```
ТЕСТ RUP-001 «Период обучения»
Тип: DET      Severity: CRITICAL
Контекст: по каждой дисциплине
Тогда:   assertEquals(РУП.курс, силлабус.курс) И assertEquals(РУП.семестры, силлабус.семестры)
```

```
ТЕСТ RUP-002 «Кредиты и ECTS»
Тип: DET      Severity: CRITICAL
Контекст: по каждой дисциплине
Тогда:   assertEquals(РУП.кредиты, силлабус.кредиты) И assertEquals(РУП.ects, силлабус.ects)
```

```
ТЕСТ RUP-003 «Общий объём и разбивка часов»
Тип: DET      Severity: CRITICAL
Контекст: по каждой дисциплине
Тогда:   по каждой позиции {всего, лекции, практические, СРОП, СРО, клин. база}:
         assertEquals(РУП.часы[x], силлабус.раздел2.часы[x])
При FAIL: таблица расхождений «позиция / в РУП / в силлабусе»
```

```
ТЕСТ RUP-004 «Кафедра/НИИ»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Тогда:   assertEquals(РУП.кафедра, силлабус.кафедра)
```

```
ТЕСТ RUP-005 «Связь модуль → дисциплины»
Тип: DET      Severity: CRITICAL
Контекст: только module
Тогда:   множество дисциплин силлабуса == множеству дисциплин модуля по РУП/отделу
При FAIL: перечислить лишние/недостающие дисциплины
```

```
ТЕСТ RUP-006 «Пререквизиты и постреквизиты»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Тогда:   assertEquals(РУП.пререквизиты, силлабус.пререквизиты)
         И assertEquals(РУП.постреквизиты, силлабус.постреквизиты)
```

## 6. Сьют INT — Внутренняя согласованность

Агент: **Аудитор согласованности**. Не требует внешних фикстур — силлабус сверяется сам с собой. Это самый большой сьют; группы: коды и нумерация (010-е), часы (020-е), покрытие (030-е), оценивание (040-е), прочие связи (050-е).

### Коды и нумерация

```
ТЕСТ INT-010 «Сквозная нумерация РО»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Дано:    раздел 5, коды РО
Тогда:   коды образуют последовательность РО1..РОn без пропусков и дубликатов;
         у каждого РО непустая формулировка и непустой маппинг на программные РО
```

```
ТЕСТ INT-011 «Сквозная нумерация ПН»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Тогда:   коды ПН1..ПНm без пропусков и дубликатов; формулировки непустые
```

```
ТЕСТ INT-012 «Ссылочная целостность кодов в тематическом плане»
Тип: DET      Severity: CRITICAL
Контекст: по каждой дисциплине
Дано:    раздел 7 (тематический план), разделы 5 (РО) и 6 (ПН)
Тогда:   каждый код РО из темплана ∈ {коды раздела 5}
         И каждый код ПН из темплана ∈ {коды раздела 6}
При FAIL: строка темплана + «битый» код
```

### Часы и кредиты

```
ТЕСТ INT-020 «Арифметика часов в описании»
Тип: DET      Severity: CRITICAL
Контекст: по каждой дисциплине
Дано:    раздел 2: total, lecture, practical, srop, sro, clinicalPractice
Тогда:   assertEquals(total, lecture + practical + srop + sro + clinicalPractice)
         И аудиторные = lecture + practical; внеаудиторные = srop + sro
```

```
ТЕСТ INT-021 «Темплан сходится с описанием по часам»
Тип: DET      Severity: CRITICAL
Контекст: по каждой дисциплине
Дано:    раздел 7 (колонки часов) и раздел 2
Когда:   считаем Σ по каждому виду: лекции, практические, СРОП, СРО
Тогда:   по каждому виду: assertEquals(раздел2.часы[вид], Σтемплан[вид])
При FAIL: «вид часов / в разделе 2 / сумма по темплану / номера строк»
```

```
ТЕСТ INT-022 «Кредиты модуля = сумма кредитов дисциплин»
Тип: DET      Severity: CRITICAL
Контекст: только module
Тогда:   assertEquals(модуль.кредиты, Σ дисциплины.кредиты);
         титульная цифра кредитов совпадает с этой суммой
```

### Покрытие (alignment)

```
ТЕСТ INT-030 «Каждый РО достигается темпланом»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Тогда:   для каждого РО раздела 5 существует ≥1 строка темплана с этим кодом
При FAIL: перечень «висячих» РО (заявлены, но ничем не достигаются)
```

```
ТЕСТ INT-031 «Каждый ПН отрабатывается темпланом»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Тогда:   для каждого ПН раздела 6 существует ≥1 строка темплана с этим кодом
```

```
ТЕСТ INT-032 «Каждая строка темплана заполнена полностью»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Тогда:   в каждой строке: тема ≠ пусто, код РО ≠ пусто, часы > 0,
         методы обучения ≠ пусто, методы/инструменты оценивания ≠ пусто
```

```
ТЕСТ INT-033 «Связка тема → РО → метод → оценивание логична»
Тип: LLM      Severity: MAJOR
Контекст: по каждой дисциплине
Дано:    строки темплана
Когда:   агент оценивает: тема реально работает на указанный РО/ПН;
         метод обучения уместен; инструмент оценивания способен измерить этот РО
Тогда:   assertCoherent(тема, РО, методОбучения, методОценивания)
При FAIL: цитата строки + объяснение, что именно не согласуется
```

```
ТЕСТ INT-034 «Единый метод оценивания внутри одного РО»
Тип: DET      Severity: MAJOR
Контекст: по каждой дисциплине
Дано:    строки темплана, сгруппированные по коду РО

При FAIL: код РО + перечень конфликтующих методов по строкам
```

## 7. Сьют TXT — Качество текста

Агент: **Редактор-вычитчик** (полностью LLM, с цитатами-доказательствами).

```
ТЕСТ TXT-001 «Нет копипаст-ошибок из чужой дисциплины»
Тип: LLM      Severity: MAJOR
Контекст: по каждой дисциплине
Когда:   агент ищет термины/органы/нозологии, чужеродные предмету дисциплины,
         в темах, чек-листах, OSCE-станциях, вопросах КИС
         (классический случай: «сердечно-сосудистой» в OSCE по дыхательной системе)
Тогда:   assertEmpty(чужеродныеФрагменты)
При FAIL: цитата + раздел + предполагаемый источник копипаста
```

```
ТЕСТ TXT-002 «Нет заглушек и черновых пометок»
Тип: DET+LLM      Severity: CRITICAL
Контекст: силлабус в целом
Когда:   DET: поиск маркеров («ВСТАВИТЬ», «TODO», «???», «XXX», «[…]», поля шаблона);
         LLM: незавершённые фразы, оборванные таблицы
Тогда:   assertEmpty(найденныеЗаглушки)
```

```
ТЕСТ TXT-003 «Язык контента — русский»
Тип: DET      Severity: MINOR
Контекст: силлабус в целом
Тогда:   основной текст на русском; иноязычные вкрапления — только принятые
         термины (OSCE, MEQ, CBL…) из глоссария
```

```
ТЕСТ TXT-004 «РО сформулированы диагностично»
Тип: LLM      Severity: MINOR (WARN)
Контекст: по каждой дисциплине
Когда:   агент проверяет формулировки РО: активный измеримый глагол
         (применяет, интерпретирует, демонстрирует…), а не «знать/понимать/уметь»
         без предмета; формулировка проверяема выбранным инструментом оценивания
Тогда:   несоответствия → WARN с предложением переформулировки
```

```
ТЕСТ TXT-005 «Внутренние повторы и противоречия»
Тип: LLM      Severity: MAJOR
Контекст: силлабус в целом
Когда:   агент ищет взаимоисключающие утверждения (разные проценты ТК/ПА в разных
         местах, разные даты, разное число тем) и дословные дубли тем
Тогда:   assertEmpty(противоречия)
```

## 8. Агент-арбитр и сводный отчёт

Арбитр не выполняет проверок. Его задачи:

1. **Дедупликация** — одно первопричинное расхождение не должно порождать каскад замечаний (если часы в разделе 2 не совпали с РУП, то INT-021 закономерно тоже упадёт — арбитр связывает их в одну корневую причину).
2. **Приоритизация** — сортировка: CRITICAL → MAJOR → MINOR → WARN.
3. **Сведение `NEEDS_HUMAN`** — отдельный блок «на ручную экспертизу» с цитатами.
4. **Итоговый статус** по правилу из п. 1.3.

Формат записи в отчёте (одна на каждый не-PASS вердикт):

```json
{
  "testId": "INT-021",
  "verdict": "FAIL",
  "severity": "CRITICAL",
  "discipline": "Принципы диагностики и лечения заболеваний ССС и дыхательной системы",
  "location": "Раздел 7, тематический план, колонка «СРО»",
  "expected": "СРО: 60 ч (раздел 2)",
  "actual": "Σ СРО по темплану = 54 ч",
  "evidence": "строки 12–18 темплана",
  "recommendation": "Привести часы СРО в темплане к 60 ч либо исправить раздел 2 по РУП",
  "confidence": 1.0
}
```

Человекочитаемый акт повторяет структуру действующего 20-этапного алгоритма (этап → замечания), чтобы экспертам не менять привычную форму.

## 10. Порядок прогона

1. **Экстракция** (`F-SYL` → JSON по модели `src/types/syllabus.ts`). Ошибка экстракции раздела → все тесты этого раздела получают `SKIP` с причиной, а сам факт — `FAIL` теста STR-002.
2. **Быстрые блокировки**: STR-001, STR-002, TXT-002 — если упали с CRITICAL, остальное прогоняется, но в отчёте сразу помечается «силлабус не готов к экспертизе».
3. **Параллельный прогон** сьютов FMT, OP, RUP, INT, TXT (независимы друг от друга).
4. **Арбитр** — сводный отчёт.
